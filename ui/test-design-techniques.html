💡 Key principle: These are all black-box techniques — you design tests based on what the system should do, not how the code works internally. They complement white-box techniques (unit tests, code coverage).

Why use these techniques?

Without techniques (random testing)

  • Tests are redundant — many cover the same logic
  • Easy to miss critical edge cases
  • Hard to know when you've tested "enough"
  • No systematic coverage of requirements
  • Difficult to justify test selection to stakeholders

With techniques (systematic testing)

  • Minimal tests, maximum coverage
  • Edge cases are found by design, not luck
  • Clear stopping criteria (all partitions covered)
  • Every test traces back to a requirement
  • Repeatable and justifiable to the team

01 Equivalence Partitioning

Divide all possible input values into groups (partitions) where the system behaves the same way. Test one value from each partition — because if one value in a partition passes, all values in that partition should behave the same.

Core Concept

EP assumes that inputs within the same partition will be processed identically. So instead of testing every possible value, you pick one representative value from each partition. This drastically reduces the number of tests while maintaining good coverage.

Example: Age field for a cinema discount system

Rule: Children (0–12) get 50% discount. Adults (13–64) pay full price. Seniors (65–120) get 30% discount. Invalid ages are rejected.

EP Rules

Valid partitions

  • Inputs the system should accept
  • At least one test per valid partition
  • Try to use a "typical" mid-range value
  • Covers the happy path scenarios

Invalid partitions

  • Inputs the system should reject
  • At least one test per invalid partition
  • Test that correct error messages appear
  • Never combine multiple invalid values in one test
⚠️ Common mistake: Testing only the happy path (valid partition). Always test all invalid partitions too — bugs often hide in error-handling code.

02 Boundary Value Analysis

Bugs cluster at the edges of input ranges. BVA tests the values exactly at, just below, and just above each boundary. It extends EP by focusing specifically on the most fault-prone values.

Why boundaries are risky

Developers often make "off-by-one" errors. A condition written as > when it should be >= only fails at the exact boundary. BVA is designed to catch exactly these bugs.

// Bug: should be >= 18 but developer wrote > if (age > 18) { // fails at exactly 18! grantAccess(); } // BVA tests age 17, 18, 19 — catches this bug at 18

03 Decision Table Testing

A systematic way to test all combinations of business rules. Decision tables map every combination of conditions (inputs) to a specific action (output), making it impossible to miss a combination.

Structure

A decision table has:

  • Conditions (rows at top) — the inputs / business rules (Y/N or specific values)
  • Actions (rows at bottom) — what the system does for each combination
  • Columns (rules) — each column is one combination = one test case

Example: E-Commerce Discount System

Rules: Logged-in users get 10% discount. Cart over $100 gives free shipping. Members get an extra 5% off. All combinations:

Click R1–R8 in the table to highlight a column. The number of test cases for N binary conditions is always 2^N. Here: 2³ = 8 rules.

Click a rule number (R1–R8) in the table header to see that test scenario

How to build a decision table

  1. List all conditions (inputs / rules). Each condition has Y/N values.
  2. Calculate total rules: 2^(number of conditions)
  3. Fill in all combinations (like binary counting: YYY, YYN, YNY, YNN…)
  4. For each combination, determine the action (what the system should do)
  5. Each column becomes one test case
  6. Optionally merge columns with identical actions (called "collapsing")
⚠️ With many conditions, the table grows exponentially: 5 conditions = 32 rules. Use collapsing (merging rules with same outcome) to reduce the table size.

04 State Transition Testing

Systems can exist in different states, and events trigger transitions between them. STT tests that the system correctly transitions from one state to another — and correctly handles invalid transitions.

Key Terms

State

A stable condition the system is in at a given time. Example: an order can be Pending, Paid, Shipped, or Delivered.

Event / Trigger

An input or action that causes a state change. Example: "Payment received" moves an order from Pending → Paid.

Transition

The movement from one state to another in response to an event, possibly with a guard condition and an action.

What to test in state transition testing

Valid transitions (happy path)

  • Each valid transition occurs correctly
  • Correct new state reached after event
  • Correct action executed on transition
  • Complete valid sequence (Pending→Paid→Shipped→Delivered)
  • All loops (e.g., retry payment multiple times)

Invalid transitions (negative tests)

  • Event that shouldn't work in current state
  • Example: "Ship" an order that hasn't been paid yet
  • Example: "Cancel" an already-delivered order
  • System should reject and stay in current state
  • Correct error message shown

05 Use Case Testing

Tests derived from use cases — descriptions of how a user interacts with the system to achieve a goal. Each use case has a main flow (happy path) and alternative/exception flows (what can go wrong).

Use Case Structure

Use Case document contains:

  • Use Case ID + Name — unique identifier
  • Actor — who initiates the use case
  • Preconditions — what must be true before
  • Main Flow — step-by-step happy path
  • Alternative Flows — valid variations
  • Exception Flows — error scenarios
  • Postconditions — state after success

Test cases derived from each:

  • One test for the full main flow
  • One test per alternative flow
  • One test per exception flow
  • One test for precondition violation
  • One test verifying postconditions
  • One test for each actor variation

Combining techniques — real project strategy

A practical approach for a login module:

  1. Use Case Testing — define the overall "Login" use case with main flow + exception flows (wrong password, locked account, forgot password)
  2. Equivalence Partitioning — partition email field (valid email / invalid format / empty) and password field (valid / too short / too long)
  3. Boundary Value Analysis — test password at exactly 8 chars, 7 chars, 9 chars, 20 chars, 21 chars
  4. Decision Table — combine remember-me + valid credentials + account locked → what happens in each combination?
  5. State Transition — model login attempt states: Active → 1 fail → 2 fails → Locked → Unlock request → Active
💡 Coverage pyramid: Use EP/BVA for field-level unit tests, Decision Tables for business rule integration tests, Use Cases for E2E acceptance tests, and State Transition for workflow regression tests.
