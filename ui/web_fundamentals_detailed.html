🖥️ Client-Server Architecture

The model that defines how the entire internet works

What is it?

The client-server model is an architecture where two types of computers communicate over a network: clients that request services, and servers that provide them. This is the foundation of the internet — every website visit, API call, or app you use follows this pattern.

🧑 CLIENT
Browser / Mobile App
Desktop App / CLI tool
① HTTP Request →
━━━━━━
← ② HTTP Response
🖥️ SERVER
Web Server / API Server
Database Server / CDN
All communication happens over the Internet using standard protocols

The Client

A client is any software that initiates a request. It is always the one that starts the conversation — servers never reach out to clients on their own (in standard HTTP).

  • Web Browser — Chrome, Firefox, Safari (the most common client)
  • Mobile App — Instagram app, banking app (talking to backend servers)
  • CLI tools — curl, wget — make HTTP requests from the terminal
  • Your own code — a Python requests.get() call is also a client

The Server

A server is software that listens for incoming requests and responds to them. It runs continuously, waiting for clients to connect.

  • Web Server — Nginx, Apache — serves HTML, CSS, JS files
  • Application Server — Node.js, Django, Spring — runs business logic
  • Database Server — PostgreSQL, MongoDB — stores and retrieves data
  • CDN Server — Cloudflare, AWS CloudFront — serves static files from locations near the user

How a Real Request Works (Step by Step)

When you type https://example.com/products and press Enter:

  1. DNS Lookup — Browser converts example.com to an IP address like 93.184.216.34
  2. TCP Connection — Browser establishes a connection to that IP on port 443
  3. TLS Handshake — Browser and server set up encryption (for HTTPS)
  4. HTTP Request — Browser sends GET /products HTTP/1.1
  5. Server Processing — Server reads the request, fetches data, builds a response
  6. HTTP Response — Server sends back 200 OK + HTML content
  7. Rendering — Browser parses HTML, requests CSS/JS/images (more requests!)
  8. Page displayed — You see the finished page

Ports

Servers listen on specific port numbers — like doors in a building. Each port handles a different type of service:

Stateless Nature of HTTP

HTTP is stateless — the server does NOT remember previous requests from the same client. Every request is treated as brand new. This is why websites use cookies and session tokens — to re-identify you on each request.

💡 Analogy: Imagine calling a call center where the agent has zero memory of your previous calls. Every time you call, you must reintroduce yourself. Websites solve this with cookies — like giving you a membership card to show each time.

Peer-to-Peer vs Client-Server

Client-Server

  • Central server handles everything
  • Easier to manage and secure
  • Server can be a bottleneck
  • Used by: websites, APIs, apps

Peer-to-Peer (P2P)

  • Every node is both client and server
  • No central authority
  • More resilient, harder to control
  • Used by: BitTorrent, blockchain

📡 TCP/IP In Depth

The plumbing of the internet — how data actually moves

What is TCP/IP?

TCP/IP is not a single protocol but a suite of protocols working together. It defines how data is broken into pieces, addressed, routed across the internet, and reassembled at the destination. Without it, the internet as we know it wouldn't exist.

  • IP (Internet Protocol) — Handles addressing and routing. Gives every device a unique address and routes packets from source to destination.
  • TCP (Transmission Control Protocol) — Handles reliable, ordered delivery. Makes sure packets arrive, are in order, and asks for retransmission if any are lost.

The 4-Layer TCP/IP Model

Data travels through 4 conceptual layers. When you send data, it moves down the layers (adding headers). When received, it moves up (stripping headers).

IP Addresses

Every device on a network gets an IP address. There are two versions:

IPv4

32-bit addresses, written as 4 numbers 0–255 separated by dots.

192.168.1.100
8.8.8.8 (Google DNS)
127.0.0.1 (localhost)

~4.3 billion possible addresses — we ran out!

IPv6

128-bit addresses, written in hexadecimal groups.

2001:0db8:85a3::8a2e
:0370:7334

340 undecillion addresses — essentially unlimited

TCP vs UDP

TCP — Reliable

  • Guarantees delivery of all packets
  • Guarantees correct order
  • Slower — more overhead
  • Used for: HTTP, HTTPS, email, file transfer

UDP — Fast

  • Fire-and-forget — no delivery guarantee
  • No ordering guarantee
  • Faster — less overhead
  • Used for: video calls, gaming, live streaming

🔒 HTTP & HTTPS In Depth

The protocol powering every web interaction

What is HTTP?

HTTP (HyperText Transfer Protocol) is an application-layer protocol that defines how messages are formatted and transmitted between a client and server. Every time you visit a webpage, HTTP is the language being spoken.

  • Text-based — HTTP messages are human-readable plain text
  • Request-Response — always one request, one response
  • Stateless — each request is independent, server has no memory of past requests
  • Application layer — sits on top of TCP/IP

HTTP Versions

VersionYearKey Feature
HTTP/1.01996One request per TCP connection — very slow
HTTP/1.11997Persistent connections (keep-alive), chunked transfer — still widely used
HTTP/22015Multiplexing (multiple requests over one connection), binary protocol, header compression
HTTP/32022Built on QUIC (UDP-based), faster connection setup, better for mobile/lossy networks

What is HTTPS?

HTTPS = HTTP + TLS (Transport Layer Security). TLS is a cryptographic protocol that wraps the HTTP message in an encrypted envelope. The server proves its identity with a digital certificate, and both sides agree on an encryption key.

🔓 HTTP

  • Data in plain text
  • Anyone on same network can read it
  • No identity verification of server
  • URL starts with http://
  • Port 80

🔒 HTTPS

  • Data is encrypted (TLS)
  • Interceptors see random bytes
  • Server identity verified by certificate
  • URL starts with https://
  • Port 443

The TLS Handshake (How HTTPS Establishes Security)

  1. Client Hello — Client says "I support these TLS versions and encryption algorithms"
  2. Server Hello — Server picks the best algorithm and sends its SSL certificate
  3. Certificate Verification — Client checks the certificate is valid and signed by a trusted Certificate Authority (CA) (like Let's Encrypt, DigiCert)
  4. Key Exchange — Client and server perform math magic to agree on a shared secret key without ever sending the key itself (Diffie-Hellman)
  5. Finished — Both confirm they're using the same key. All future messages are encrypted.

SSL Certificates

A certificate contains:

  • The domain name it's issued for (e.g., example.com)
  • The server's public key
  • The issuing Certificate Authority (CA)
  • Expiry date (usually 90 days to 1 year)
  • Digital signature proving the CA verified it
⚠️ If you see a "Your connection is not private" warning, the certificate is expired, invalid, or the domain doesn't match. Do NOT proceed if entering sensitive data.

HTTP vs HTTPS — What Gets Encrypted?

  • ✅ Encrypted: headers, body, URL path and query string, cookies
  • ❌ NOT encrypted: the destination IP address and domain name (SNI — visible to ISPs)
🔴 Never send passwords, tokens, or credit card info over HTTP. It's visible to anyone on your Wi-Fi network — coffee shop, airport, etc.

📨 HTTP Requests & Responses

The exact structure of every web message

HTTP Request Structure

Every HTTP request has 3 parts:

━━━ 1. REQUEST LINE ━━━

━━━ 2. HEADERS ━━━
Host: api.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGci...
Content-Length: 47

━━━ 3. BODY (blank line separates headers from body) ━━━
{"name": "Alice", "email": "alice@ex.com"}

HTTP Response Structure

━━━ 1. STATUS LINE ━━━
HTTP/1.1 201 Created

━━━ 2. HEADERS ━━━
Content-Type: application/json
Content-Length: 89
Location: /api/users/1042
Date: Fri, 17 Apr 2026 10:30:00 GMT

━━━ 3. BODY ━━━
{"id": 1042, "name": "Alice", "email": "alice@ex.com", "created": true}

🚦 HTTP Status Codes — Complete Reference

Every number tells a story about what happened to your request

Status codes are 3-digit numbers in every HTTP response. The first digit defines the category. There are 5 categories:

2xx
Success
3xx
Redirect
4xx
Client Error
5xx
Server Error
1xx
Informational

✅ 2xx — Success

200

OK

The request succeeded. The response body contains what was requested. Default success for GET requests.

Example: GET /users/1 → 200 OK with user data in body
201

Created

A new resource was successfully created. Usually returned after a POST. Often includes a Location header pointing to the new resource.

Example: POST /users → 201 Created + Location: /users/42
204

No Content

Request succeeded but there's nothing to return in the body. Common for DELETE or PUT.

Example: DELETE /posts/5 → 204 No Content

↪️ 3xx — Redirection

301

Moved Permanently

The resource has permanently moved to a new URL (in the Location header). Browsers update bookmarks. Search engines transfer SEO ranking to new URL.

Example: Migrating from http:// to https://
302

Found (Temporary Redirect)

Temporarily at a different URL. Browser uses the original URL for future requests.

Example: Redirect to login page while not authenticated
304

Not Modified

Used with caching. Client sends If-None-Match or If-Modified-Since, server says "nothing changed, use your cached version." Saves bandwidth.

❌ 4xx — Client Errors (You Did Something Wrong)

400

Bad Request

The server can't understand your request due to invalid syntax, missing required fields, or malformed JSON. Fix your request before retrying.

401

Unauthorized

You must authenticate first. The response usually includes a WWW-Authenticate header explaining how. Despite the name, it really means "unauthenticated."

Fix: include a valid Authorization header (token, API key)
403

Forbidden

You're authenticated but don't have permission. The server understood your request but refuses to fulfill it. Different from 401 — you can't fix it by logging in.

Example: A regular user trying to access an admin panel
404

Not Found

The requested resource doesn't exist at this URL. Either the URL is wrong, the resource was deleted, or it never existed. Also used intentionally to hide existence of private resources.

405

Method Not Allowed

The resource exists but doesn't support that HTTP method. Server includes an Allow header listing valid methods.

Example: Sending DELETE to an endpoint that only accepts GET
409

Conflict

Request conflicts with current state of the resource.

Example: Trying to register an email that already exists
422

Unprocessable Entity

Syntactically correct but semantically wrong. Used in APIs when data is valid JSON but fails business logic validation.

Example: age field contains -5 (invalid value)
429

Too Many Requests

You've been rate-limited. The server includes a Retry-After header telling you when to try again. APIs use this to prevent abuse.

💥 5xx — Server Errors (Server's Fault)

500

Internal Server Error

Generic server crash. An unhandled exception, bug, or unexpected condition on the server. Check server logs. The client did nothing wrong.

502

Bad Gateway

A proxy/load balancer got an invalid response from the upstream server. Common in microservice architectures when one service returns garbage.

503

Service Unavailable

Server is temporarily down — overloaded or under maintenance. Try again later. Often includes a Retry-After header.

504

Gateway Timeout

A proxy waited too long for the upstream server to respond. The upstream server is too slow or unreachable.

✅ Quick memory rule: 2xx = It worked · 3xx = Go somewhere else · 4xx = You messed up · 5xx = Server messed up

📋 HTTP Headers — Deep Dive

The metadata envelope around every HTTP message

What Are Headers?

Headers are key-value pairs sent with every HTTP request and response. They carry metadata — information about the message — such as content type, authentication, caching instructions, encoding, and more. Headers are invisible to end users but essential for developers.

Header-Name: header-value
Note: header names are case-insensitive, values may be case-sensitive

CORS Headers

CORS (Cross-Origin Resource Sharing) is a browser security mechanism. By default, a webpage at myapp.com cannot make API calls to api.other.com. The server must explicitly allow it with CORS headers.

━ SERVER RESPONSE HEADERS (allow cross-origin access) ━
Access-Control-Allow-Origin: https://myapp.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400
💡 DevTools Tip: Open Chrome DevTools (F12) → Network tab → click any request → you can see all request and response headers. This is how developers debug API issues in real life.
