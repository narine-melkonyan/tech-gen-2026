Request Lifecycle

Everything that happens between clicking a link and seeing the page. Click each step to expand the full explanation.

💡 DevTools shortcut: Press F12 → Network tab → check "Preserve log" → navigate around. You can see every step of this cycle in real time for every request your browser makes.

SPA vs MPA Navigation

Multi-Page App (MPA)

Every link click triggers a full HTTP request cycle. The browser fetches a new HTML page from the server. Simple, reliable, great for SEO. Used by: traditional websites, e-commerce, blogs.

/* Full page load on every nav */ GET /products → 200 OK → new HTML GET /cart → 200 OK → new HTML

Single-Page App (SPA)

JS intercepts link clicks. Only data (JSON) is fetched, the DOM is updated in place. Faster after first load, but requires careful handling of browser history, back button, and deep links.

/* JS router handles navigation */ // No HTTP request — just re-render router.push('/products'); GET /api/products → 200 JSON only

DNS in Detail

DNS (Domain Name System) converts a human-readable domain like example.com into a machine-readable IP address like 93.184.216.34. This lookup happens in layers, each with its own cache:

LayerCache TTLWhat happens
Browser DNS cache~60sFastest — browser remembers recent lookups
OS DNS cachevariesSecond check — OS-level resolver
ISP ResolvervariesYour internet provider's DNS server
Root DNS serverspermanent13 root servers that know where .com, .org, etc. live
Authoritative DNSset by ownerThe actual DNS record with your IP address
⚠️ Testing tip: Always clear your DNS cache when testing domain changes. In Chrome: chrome://net-internals/#dns → click "Clear host cache".

Navigation Demo

A working simulation of a real application navbar with all navigation states. Click each link to see how it behaves and what testers should check.

MyApp Home Products Dashboard Docs ↗ About Deleted

🏠 Home Page

Welcome! Notice the active state — "Home" has a bottom border highlight. Testers verify: active class updates on navigation, the correct item is highlighted, and no two items are active simultaneously.

📦 Products Page

Product listing. Testers check: page title updates in the browser tab, the URL changes to /products, the back button returns to Home, and deep-linking http://localhost/products loads this page directly.

🔒 Dashboard (protected route)

This route requires authentication. If not logged in, clicking Dashboard should redirect to /login?redirect=/dashboard, NOT show a blank page or a 500 error. Testers verify the redirect works AND that after logging in you land here, not on the homepage.

↗ External Link — opened in new tab

External links must use target="_blank" AND rel="noopener noreferrer". Without noopener, the opened page can access window.opener and redirect your app to a phishing site. Test: original tab still shows MyApp after click.

🔴 Security risk: <a href="https://evil.com" target="_blank"> — missing rel attribute

ℹ️ About Page

Static content page. Check: heading hierarchy (H1→H2 in correct order), meta description present, canonical URL set, no broken images, and all in-page anchor links (#section) scroll to the right element.

💥 404 — Page Not Found

This page was deleted. Testers verify: the server actually returns HTTP 404 (check DevTools Network tab — not 200!), a friendly error page is shown (not a white screen), there's a link back to Home, and the page title says "Not Found".

Common bug: SPA returns 200 OK for this URL even though the page doesn't exist — because the server always serves index.html. The HTTP status must be 404.

What testers check on every navigation

Visual checks

  • Active nav item is correctly highlighted
  • No two items highlighted at once
  • Page title in browser tab updates
  • Breadcrumbs (if any) reflect current page
  • Scroll position resets to top on navigation
  • Loading state shown during navigation

Functional checks

  • URL changes to match the page
  • Browser back/forward buttons work
  • Pasting the URL directly loads correct page
  • Refresh keeps you on the same page
  • Protected routes redirect to login
  • 404 page shown for unknown URLs

Browser History API (for SPAs)

SPAs manage navigation manually using the History API. Understanding this helps testers know what to look for.

// Push a new entry to browser history (adds to back stack) window.history.pushState({page: 'products'}, '', '/products'); // Replace current entry (doesn't add to back stack) window.history.replaceState({page: 'home'}, '', '/'); // Listen for back/forward button clicks window.addEventListener('popstate', (event) => { // re-render based on event.state or window.location renderPage(window.location.pathname); });

Testing anchor (#) links

Anchor links are deceptively tricky, especially in SPAs where content loads asynchronously.

// Problem: content isn't loaded yet when scroll fires document.querySelector('#section')?.scrollIntoView(); // null = no scroll // Fix: scroll AFTER content is rendered setTimeout(() => { document.querySelector(window.location.hash) ?.scrollIntoView({ behavior: 'smooth' }); }, 100); // give the DOM time to render

API Status Codes

Every HTTP response starts with a status code. Click any row to see when it's returned, what headers it sends, and how to test it in your Node.js project.

✅ Quick memory rule: 2xx = it worked · 3xx = go somewhere else · 4xx = your fault · 5xx = server's fault

Status codes in your Express API

// Always set status BEFORE sending response res.status(201).json({ success: true, data: newProduct }); // Common mistake — defaults to 200, even for errors! if (!product) { return res.json({ success: false, message: 'Not found' }); // BUG: sends 200 } // Correct if (!product) { return res.status(404).json({ success: false, message: 'Not found' }); }
