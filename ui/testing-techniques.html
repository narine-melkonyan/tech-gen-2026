Static Testing

Testing without running the software. You examine artifacts — code, requirements, design documents, test plans — to find defects early. The earlier a defect is found, the cheaper it is to fix.

Two main types of static testing

Reviews

Human-led examination of documents. A team of people reads through a work product and finds issues through discussion and inspection.

  • Informal review / peer review
  • Walkthrough
  • Technical review
  • Formal inspection (Fagan)

Static Analysis

Automated tools that analyze source code without running it. Find bugs, code smells, security vulnerabilities, and style issues.

  • Linters (ESLint, Pylint)
  • SAST tools (SonarQube, Checkmarx)
  • Type checkers (TypeScript, mypy)
  • Dependency scanners (npm audit)

Review types — from informal to formal

1

Informal Review / Peer Review

Author asks a colleague to look at their work. No defined process, no documentation required. Fast and lightweight. Common as a GitHub Pull Request. Best for: day-to-day code changes, small features.

2

Walkthrough

Author leads a meeting where they explain their work to peers step by step. The audience asks questions and suggests improvements. Author retains full control — no formal defect recording required. Best for: sharing knowledge, onboarding, design decisions.

3

Technical Review

A structured meeting led by a trained moderator. Reviewers prepare in advance using checklists. Defects are logged and tracked. More formal than walkthrough. Best for: architecture documents, API designs, complex modules.

4

Formal Inspection (Fagan Inspection)

The most rigorous review type. Defined roles: Moderator, Author, Reviewers, Recorder. Entry/exit criteria. Defects tracked and categorized. Metrics collected. Proven to find 60–90% of defects before testing begins. Best for: safety-critical systems, high-value modules.

Roles in a formal inspection

RoleResponsibility
ModeratorFacilitates the inspection, enforces process, ensures objectivity, tracks defects
AuthorCreated the work product being reviewed. Answers questions but does NOT defend their work
Reviewer / InspectorStudies the work product before the meeting, identifies defects using checklists
Recorder / ScribeDocuments all defects found during the review meeting
Reader(Optional) Reads the document aloud during the meeting so all reviewers follow the same pace

Dynamic Testing

Testing by actually running the software with real inputs and observing real outputs. This is the most widely practiced form of testing — from manual exploratory testing to automated test suites.

What makes it "dynamic"

Dynamic testing requires the software to be executable. You provide inputs, the system runs, and you compare the actual output against the expected output. Any difference is a potential defect.

// Dynamic test: we actually CALL the function const result = calculateDiscount({ age: 70, memberType: 'senior' }); // Then COMPARE actual vs expected: expect(result).toBe(0.30); // 30% discount expected // If result is 0.10 instead of 0.30, that's a defect!

Testing levels (where in the lifecycle)

U

Unit Testing

Test individual functions, methods, or classes in isolation. Dependencies are mocked. Fastest feedback. Done by developers. Tools: Jest, Mocha, JUnit, pytest. Finds: logic errors in single functions.

I

Integration Testing

Test how modules interact — API + database, microservice A + microservice B. Real connections, not mocks. Tools: Supertest, Postman, REST-assured. Finds: interface mismatches, data format errors, auth failures between services.

S

System Testing

Test the entire system end-to-end in an environment close to production. All components integrated. Tests based on requirements. Tools: Playwright, Cypress, Selenium. Finds: requirement gaps, end-to-end workflow failures.

A

Acceptance Testing (UAT)

Business stakeholders or actual users verify the system meets business requirements. Often the "sign-off" before production deployment. Finds: requirements misunderstandings, missing business-critical features.

Testing types (what you're checking)

Functional testing

  • Functional — does it do what it should?
  • Regression — did new code break existing features?
  • Smoke — do critical paths work after a build?
  • Sanity — is this specific fix/change working?
  • Exploratory — unscripted, curiosity-driven

Non-functional testing

  • Performance — speed, throughput, response time
  • Load — behavior under expected traffic
  • Stress — behavior beyond normal limits
  • Security — auth, injection, XSS, CSRF
  • Usability — is the UI easy to use?
  • Accessibility — WCAG compliance

Test pyramid — balance of dynamic tests

The test pyramid describes the ideal ratio of dynamic tests. More unit tests, fewer E2E tests — for speed and maintainability.

E2E Tests
Few, slow, expensive
Integration Tests
Medium number, medium speed
Unit Tests
Many, fast, cheap — the foundation
⚠️ Anti-pattern — inverted pyramid ("ice cream cone"): Teams that skip unit tests and rely mostly on E2E tests have slow, flaky, expensive test suites that break constantly and are hard to debug. Build from the bottom up.

Understanding when to apply each approach — and how they complement each other — is what separates a junior tester from a senior QA engineer.

✅ Best practice: Apply static testing first (reviews + linting in CI). Then dynamic testing. You'll find more bugs earlier and at a fraction of the cost of finding them later.

Black Box Testing

You test the system from the outside — like a real user. You provide inputs and check outputs, without any knowledge of the internal code, architecture, or implementation. The system is a "black box."

The black box mental model

📥
Input
email + password
→→→
⬛
System
(internal code unknown)
→→→
📤
Output
200 OK / 401 Error

You only care that correct inputs produce correct outputs. You don't care HOW the code achieves that.

Black box techniques (you've already learned these!)

Data-driven techniques

  • Equivalence Partitioning — group inputs into valid/invalid classes
  • Boundary Value Analysis — test at the edges of each class
  • Decision Table Testing — test all combinations of conditions
  • Pairwise Testing — test all pairs of inputs (reduced combinations)

Experience-based techniques

  • Exploratory Testing — simultaneous test design + execution guided by intuition
  • Error Guessing — guess likely errors based on experience
  • Checklist-based — use predefined checklists of known risk areas
  • Use Case Testing — test complete user scenarios

Advantages and limitations

Advantages

  • No code knowledge needed — QA testers can write tests independently
  • Tests match the user's actual perspective
  • Unbiased — tester is not influenced by how the code was written
  • Applicable at all testing levels (unit through acceptance)
  • Great for requirement validation

Limitations

  • Cannot test internal logic paths or code branches
  • May miss internal bugs that don't affect visible outputs
  • Hard to achieve 100% path coverage
  • Test cases may be redundant (multiple tests hitting same code)
  • Cannot directly test security vulnerabilities in code
💡 When to use Black Box: Acceptance testing, system testing, regression testing, API functional testing, UI testing. Whenever you need to verify the system meets its requirements from the user's perspective.

White Box Testing

Testing with full visibility into the source code. You design tests to cover specific code paths, branches, conditions, and loops — ensuring every line of code is exercised.

Gray Box Testing

A hybrid approach: partial knowledge of the internal workings. The tester knows the architecture, database schema, API contracts, and key algorithms — but does not read line-by-line source code during testing.

What "partial knowledge" means in practice

What gray box testers know

  • Database schema and table structures
  • API contracts (endpoints, request/response formats)
  • System architecture (which services talk to which)
  • Key business logic and algorithms at a high level
  • Data flows between components

What they don't focus on

  • Line-by-line source code
  • Internal variable names
  • Specific implementation of algorithms
  • Code coverage metrics
  • Unit test implementation details

Gray box test types

  • API testing (know endpoints, not internals)
  • Database-backed functional testing
  • Integration testing between components
  • Security penetration testing
  • Regression testing with architectural knowledge

Gray box in action — API + database testing

A gray box tester testing a user registration API knows:

  • The API endpoint is POST /api/users
  • The database has a users table with a unique constraint on email
  • Passwords are hashed with bcrypt before storage
  • A verification email is sent via a mail-service microservice

With this knowledge, they design more targeted tests than a pure black box tester:

// Gray box test — knows the DB schema, tests accordingly it('duplicate email returns 409, not 500', async () => { // Gray box knowledge: DB has UNIQUE constraint on email // Black box tester might not think to test the exact error code await createUser({ email: 'test@ex.com' }); const res = await createUser({ email: 'test@ex.com' }); // duplicate expect(res.status).toBe(409); // not 500 (DB crash) or 400 (bad request) }); it('password is not stored in plaintext', async () => { // Gray box: knows the DB schema — can query it directly await createUser({ email: 't@t.com', password: 'MyPassword123' }); const row = await db.query('SELECT password FROM users WHERE email=$1', ['t@t.com']); expect(row.password).not.toBe('MyPassword123'); // must be hashed! expect(row.password).toMatch(/^\$2[ab]\$10\$/); // bcrypt format });

ScenarioBest choiceReason

🔬 Real-world strategy: Most teams use all three without calling them by name. Developers write white-box unit tests. QA writes black-box E2E tests. Senior QA or QA-in-squad writes gray-box API and integration tests. This combination gives the best coverage.
