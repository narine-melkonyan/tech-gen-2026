✅ Functional Testing for Web Applications

Testing that every feature works exactly as the requirements specify

What is Functional Testing?

Functional testing verifies that the application behaves correctly from the user's point of view. It doesn't care how the code works internally — only that inputs produce the expected outputs. It's based on what the software is supposed to do.

🔘 Testing Forms

Forms are the most critical UI element for functional testing. Every field, validation rule, and submission path must be tested.

What to test in every form:

  • Required fields — Submit without filling required fields → error messages appear
  • Valid input — Correct data → form submits successfully
  • Invalid input — Wrong format (e.g., email without @) → validation error shown
  • Boundary values — Min/max lengths, min/max numbers (e.g., age: 0, -1, 150)
  • Special characters — <script>, 'OR 1=1, emojis — must not break
  • Whitespace — Spaces-only input should be treated as empty
  • Duplicate submission — Double-click submit → should not create two records
  • Tab order — Tab key moves logically through fields
  • Autofill — Browser autofill should not break form state

🔗 Testing Links & Navigation

  • Internal links — Each link navigates to the correct page
  • External links — Open in new tab (target="_blank") and reach correct URL
  • Broken links — No 404 errors on any navigation link
  • Breadcrumbs — Correctly reflect current page hierarchy
  • Back button — Browser back navigates to previous page correctly
  • Deep links — Pasting a URL directly loads the correct page
  • Anchor links — #section links scroll to correct position
  • 404 page — Broken URLs show a helpful error page, not a crash

🖱️ Testing Buttons

  • Click action — Correct action is triggered
  • Disabled state — Disabled buttons cannot be clicked
  • Loading state — Button shows spinner/disables during async operations
  • Double-click protection — Submitting twice doesn't duplicate the operation
  • Keyboard access — Button can be triggered with Enter/Space key
  • Visual feedback — Hover, active, focus states are visually distinct
  • Correct label — Button text accurately describes its action

💡 Positive vs Negative Testing: Always test BOTH valid paths (happy path) and invalid paths (error paths). Most bugs are found in error handling, not in the happy path.

🌐 Cross-Browser Testing

Ensuring your app works correctly across all major browsers

Why Cross-Browser Testing Matters

Different browsers use different rendering engines. The same HTML/CSS/JS can behave differently across them. A feature that works perfectly in Chrome might be broken, unstyled, or missing in Safari or Firefox.

BrowserEngineMarket Share (2025)Key Concerns
🟡 ChromeBlink + V8~65%Baseline — most features work, but don't assume others match
🦊 FirefoxGecko + SpiderMonkey~4%CSS Grid differences, font rendering, some web APIs
🔷 EdgeBlink + V8 (Chromium)~13%Very similar to Chrome; IE compatibility mode is unique
🍎 SafariWebKit + JavaScriptCore~19%Most issues here — delayed feature support, iOS quirks
🟡

DevTools gold standard. Most CSS/JS works. Test others against Chrome baseline.
🦊
Firefox
Stricter CSS parsing. Different scrollbar styling. Some Date API differences.
🔷

Chromium-based so very similar to Chrome. Has IE Mode for legacy sites.
🍎
Safari
Most cross-browser bugs found here. No custom scrollbars. Flexbox quirks. Input styling.

Common Cross-Browser Issues

CSS Issues

  • Flexbox/Grid layout differences
  • CSS custom properties (variables) not supported in older browsers
  • Scrollbar styling (webkit-only)
  • Font rendering differences
  • Input/select element styling
  • gap in flexbox (Safari 14 and older)
  • CSS :focus-visible support varies

JavaScript Issues

  • ES6+ features not in older Safari
  • Date parsing format differences
  • fetch() AbortController quirks
  • Web APIs (Bluetooth, NFC) not in Safari
  • Event handling differences (e.which vs e.key)
  • IndexedDB behavior varies
  • Clipboard API requires HTTPS + user gesture

Cross-Browser Test Checklist

  • Page layout is correct in Chrome
  • Page layout is correct in Firefox
  • Page layout is correct in Edge
  • Page layout is correct in Safari
  • All fonts render correctly in all browsers
  • Forms submit correctly in all browsers
  • JavaScript functionality works in all browsers
  • Animations/transitions work in all browsers
  • Images and icons display correctly everywhere
  • Console shows no errors in any browser

test('login form works', async ({ page }) => {
  await page.goto('https://myapp.com/login');
  await page.fill('#email', 'user@test.com');
  await page.fill('#password', 'Pass123!');
  await page.click('button[type=submit]');
  await expect(page).toHaveURL(/dashboard/);
});
⚠️ Safari is the new IE. Safari on iOS (which all iOS browsers must use!) is the most common source of cross-browser bugs in 2025. Always test on real or simulated Safari/iOS.

📱 Cross-Device Testing

Testing across desktops, tablets, and mobile phones

Why Devices Differ

It's not just about screen size. Devices differ in input method (mouse vs touch), screen pixel density (retina vs standard), performance (CPU/RAM), browser (mobile Safari), and OS behavior (keyboard, status bar, safe areas).

🖥️ Desktop

  • Mouse + keyboard input
  • Hover states work
  • Large screen (1280px–2560px+)
  • Right-click context menus
  • High CPU/RAM available
  • No virtual keyboard
  • No safe area insets
  • Stable network usually

📱 Mobile

  • Touch input (finger-sized targets!)
  • No hover states
  • Small screen (360px–430px)
  • Swipe gestures
  • Limited CPU/RAM
  • Virtual keyboard pushes layout up
  • Safe area / notch insets
  • Variable network (3G/4G/5G/Wi-Fi)

Key Mobile-Specific Issues to Test

Touch & Input

  • Tap targets minimum 44×44px (Apple HIG) or 48×48dp (Material)
  • Buttons not too close together (fat-finger problem)
  • Swipe left/right doesn't conflict with browser navigation
  • Pinch-to-zoom works on images but not accidentally on forms
  • Scroll works smoothly inside scrollable containers
  • Long-press behavior (select text, context menu)

Virtual Keyboard

  • Input type email → shows @ key on keyboard
  • Input type tel → shows number pad
  • Input type number → shows numeric keyboard
  • Keyboard doesn't cover the active input
  • Form still submittable when keyboard is open
  • Content doesn't jump when keyboard opens/closes

Correct Input Types for Mobile UX

<!-- Each shows a DIFFERENT keyboard on mobile -->
<input type="email"> → @ . com keys prominent
<input type="tel"> → phone number pad
<input type="number"> → numeric keyboard
<input type="search"> → shows search/go button
<input type="url"> → / and .com keys
<input type="date"> → native date picker
<input type="password"> → masks characters

<!-- inputmode: more keyboard control -->
<input inputmode="decimal"> → decimal number pad
<input inputmode="numeric"> → 0–9 only

Device Testing Checklist

  • All tap targets are at least 44×44px
  • Virtual keyboard does not cover active input field
  • Input types show correct mobile keyboard
  • Horizontal scrolling does not appear unintentionally
  • Text is readable without pinching/zooming
  • Images load correctly on retina/HiDPI screens
  • Page works on slow 3G connection (test in DevTools)
  • No hover-dependent UI on touch devices
  • Navigation works with touch/swipe gestures
  • Safe area / notch does not cut off content (iPhone)

🎨 UI / UX Testing Basics

Testing that the interface is correct, accessible, and delightful

UI Testing vs UX Testing

UI Testing (User Interface)

Tests the visual correctness of the interface — does it look right?

  • Colors, fonts, spacing match design
  • Icons and images display correctly
  • Buttons and inputs are styled correctly
  • Layout doesn't break at any screen size
  • Visual consistency across all pages
  • Error/success/loading states are visible

UX Testing (User Experience)

Tests how easy it is to use — does it feel right?

  • User can complete tasks without confusion
  • Error messages are helpful and clear
  • Navigation is intuitive
  • Loading states communicate progress
  • Confirmation dialogs prevent accidents
  • Undo is available for destructive actions

Accessibility Testing (a11y)

Accessibility is both a legal requirement (in many countries) and good UX. About 15% of people worldwide have a disability that affects how they use the web.

  • Color contrast — Text must have at least 4.5:1 ratio against background (WCAG AA)
  • Keyboard navigation — Every interactive element must be reachable with Tab key
  • Focus indicators — Focused element must have a visible outline
  • Alt text — All images must have descriptive alt attributes
  • ARIA labels — Buttons without text must have aria-label
  • Form labels — Every input must have an associated <label>
  • Heading hierarchy — H1 → H2 → H3 in logical order, not skipping levels
  • Screen reader — Test with VoiceOver (Mac) or NVDA (Windows)
/* ❌ BAD — no label, no aria */
<button><img src="close.svg"></button>

/* ✅ GOOD — screen reader reads "Close dialog" */
<button aria-label="Close dialog"><img src="close.svg" alt=""></button>

/* ❌ BAD — input without label */
<input type="text" placeholder="Enter email">

/* ✅ GOOD — input with associated label */
<label for="email">Email address</label>
<input type="email" id="email">

UI/UX Testing Checklist

  • All colors match the design system / brand guidelines
  • Text contrast ratio meets WCAG AA (4.5:1 minimum)
  • All interactive elements are keyboard-accessible
  • Focus states are clearly visible on all focusable elements
  • All images have descriptive alt text
  • Error messages clearly explain what went wrong and how to fix
  • Loading spinners/skeletons appear during async operations
  • Success/failure feedback is provided after every user action
  • Destructive actions (delete, etc.) require confirmation
  • Page title changes to reflect current page content
  • Fonts are consistent and readable at all sizes
  • Spacing and alignment are consistent across pages

📐 Responsive Design Testing

Verifying the layout adapts correctly to every screen size

What is Responsive Design?

Responsive design means the layout fluidly adapts to any screen width — from a 320px phone to a 4K monitor — using CSS media queries, flexible grids, and fluid images.

Standard Breakpoints

BreakpointDevice TypeTypical WidthTest Focus
XSSmall phones320px – 375pxNothing overflows, text is readable
SMLarge phones376px – 480pxStandard mobile layout
MDTablets (portrait)481px – 768pxLayout switch point — sidebar or stack
LGTablets (landscape) / small laptop769px – 1024pxFull nav visible, content columns
XLDesktop1025px – 1440pxFull desktop layout
2XLWide / 4K monitor1441px+No overly stretched text, max-width applied

How to Test Responsiveness

Method 1: Chrome DevTools Device Mode

  1. Press F12 to open DevTools
  2. Click the Toggle Device Toolbar icon (📱) or press Ctrl+Shift+M
  3. Select a device from the dropdown OR enter a custom width
  4. Drag the handle to resize and watch layout change in real time
  5. Click "Responsive" and type exact pixel widths to test breakpoints

Method 2: CSS Media Query Testing

/* Mobile first approach — start with mobile styles */
.container { width: 100%; padding: 0 16px; }

/* Tablet */
@media (min-width: 768px) {
  .container { max-width: 720px; margin: 0 auto; }
  .sidebar { display: block; } /* show sidebar */
}

/* Desktop */
@media (min-width: 1024px) {
  .container { max-width: 960px; }
  .grid { grid-template-columns: 1fr 1fr 1fr; }
}

/* Test: resize browser window slowly and watch */
/* for ANY layout jump or overflow */

Responsive Testing Checklist

  • No horizontal scroll at 320px, 375px, 768px, 1024px, 1440px
  • Viewport meta tag is present in <head>
  • Images scale down and don't overflow container
  • Navigation collapses to hamburger menu on mobile
  • Text remains readable at all breakpoints (min 16px body)
  • Tables have overflow scroll wrapper on small screens
  • Multi-column layout stacks to single column on mobile
  • Form inputs are full-width on mobile
  • Modal/dialog fits within screen on all sizes
  • Footer content is accessible and not hidden on small screens

🔔 Handling Alerts, Pop-ups & Frames

Testing and automating native dialogs, modals, and iframes

Types of Alerts & Dialogs

Native Browser Alert

Created by window.alert(). Blocks the page. Only has an OK button. Cannot be styled.

alert("Hello!");

Confirm Dialog

Created by window.confirm(). Returns true (OK) or false (Cancel).

const ok = confirm("Delete?");
if (ok) deleteItem();

Prompt Dialog

Created by window.prompt(). Returns the text the user entered, or null if cancelled.

const name = prompt("Name?");

Custom Modal Dialogs (HTML/CSS)

Modern apps replace native browser dialogs with custom modals built in HTML/CSS. These can be fully styled and are preferred for UX.

<!-- Custom Modal HTML Structure -->
<div id="modal" role="dialog" aria-modal="true"
     aria-labelledby="modal-title">
  <div class="modal-overlay" onclick="closeModal()"></div>
  <div class="modal-content">
    <h2 id="modal-title">Confirm Delete</h2>
    <p>This action cannot be undone.</p>
    <button onclick="closeModal()">Cancel</button>
    <button onclick="confirmDelete()">Delete</button>
  </div>
</div>

What to Test in Modals & Pop-ups

  • Opens correctly — Modal appears when trigger is clicked
  • Closes correctly — X button, Cancel button, and Escape key all close it
  • Overlay click — Clicking outside the modal closes it (unless intentionally prevented)
  • Focus trap — Tab key should cycle only within the modal, not reach background elements
  • Focus restoration — After closing, focus returns to the element that opened the modal
  • Scroll lock — Background page should not scroll while modal is open
  • Screen reader — Modal is announced properly via ARIA role="dialog"
  • Multiple modals — Stacked modals close in the right order
  • Mobile — Modal fits screen, is scrollable if content is tall

Automating Alert Handling with Playwright

// Accept a native browser alert
page.on('dialog', async dialog => {
  console.log(dialog.message()); // get the text
  await dialog.accept(); // click OK
});

// Dismiss a confirm dialog (click Cancel)
page.on('dialog', async dialog => {
  await dialog.dismiss();
});

// Fill a prompt dialog
page.on('dialog', async dialog => {
  await dialog.accept('John Doe'); // type and click OK
});

iFrames — What They Are

An iframe (inline frame) is an HTML element that embeds another webpage inside the current one. Common uses: embedded YouTube videos, Google Maps, payment forms (Stripe, PayPal), third-party widgets.

<!-- Embedding another page inside this page -->
<iframe
  src="https://maps.google.com/embed?..."
  width="600"
  height="400"
  title="Store Location Map"
  loading="lazy"
></iframe>

What to Test in iFrames

  • Loads correctly — Embedded content appears without errors
  • Correct size — No scrollbars inside iframe, not cropped
  • Responsive — iframe scales on small screens (use aspect-ratio)
  • Fallback — Meaningful message shown if iframe fails to load
  • Accessibility — Must have a title attribute for screen readers
  • HTTPS in HTTPS — A page served over HTTPS cannot embed HTTP iframes (mixed content)
  • Cross-origin restrictions — Cannot access iframe DOM if different origin (security)
  • Keyboard navigation — Tab should be able to enter and exit the iframe

Automating iFrame Interaction with Playwright

// Locate an iframe and interact inside it
const frame = page.frameLocator('iframe[title="Payment Form"]');

// Now interact with elements INSIDE the iframe
await frame.locator('#card-number').fill('4242424242424242');
await frame.locator('#expiry').fill('12/28');
await frame.locator('#cvc').fill('123');

// Or using the older frame() method:
const iframe = page.frame({ name: 'payment' });
await iframe.fill('#card-number', '4242...');

Responsive iframe (Correct Approach)

/* ❌ BAD — fixed pixel height breaks on mobile */
iframe { width: 100%; height: 315px; }

/* ✅ GOOD — maintains 16:9 ratio at any width */
.iframe-wrapper {
  position: relative;
  width: 100%;
  aspect-ratio: 16 / 9; /* modern approach */
}
.iframe-wrapper iframe {
  position: absolute;
  inset: 0; width: 100%; height: 100%;
}
✅ Key Rule: Native browser dialogs (alert/confirm/prompt) block JavaScript execution entirely. Always handle them BEFORE the action that triggers them in automated tests.

Alerts & Frames Final Checklist

  • alert() displays the correct message
  • confirm() OK triggers the intended action
  • confirm() Cancel cancels the intended action
  • Custom modal opens and closes correctly
  • Escape key closes modal
  • Focus is trapped inside open modal
  • Background scrolling is locked when modal is open
  • iFrame loads content correctly
  • iFrame has title attribute for accessibility
  • iFrame is responsive and doesn't cause horizontal scroll
